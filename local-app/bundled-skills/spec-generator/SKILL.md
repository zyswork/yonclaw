# Spec Generator - 通用 PRD 生成器

## 概述

输入：Excel 需求表 + 项目文档目录
输出：一份完整的 PRD 文档（Markdown）

核心原则：
- **两个输入，一个输出** — 用户给路径，直接出文档，不要反复交互
- **PRD 只写"做什么"，不写"怎么做"** — 技术方案留给 AI 编码工具
- **一个文件** — 所有需求在一份 PRD 中
- **详细到可执行** — 每条需求包含用户故事、业务规则、数据模型、交互说明、异常处理、通知规则、验收标准

## 触发条件

用户提供：
- Excel 需求表路径（.xlsx / .xls / .csv）
- 项目文档/代码目录路径
- "生成 PRD"、"生成需求文档" 等意图

## 工作流程

### Step 1：解析输入

同时执行两个脚本：

```bash
# 解析 Excel 需求表
python3 <skill_dir>/scripts/parse_excel.py <excel路径>

# 扫描项目文档
python3 <skill_dir>/scripts/scan_project.py <项目目录>
```

解析完成后，向用户展示概览（一次性确认）：
- 共 N 条需求，按优先级/模块分布
- 预估总工时
- 项目技术栈摘要
- 列名映射是否正确

**用户确认后直接进入生成。** 如果用户要调整（如排除某些需求、修改工时），在这一步完成。

### Step 2：询问团队配置 + 迭代规划

概览确认后，**必须询问用户团队资源配置**：

> 📋 需要你的团队配置来做迭代规划：
> | 角色 | 人数 | 迭代天数 |
> |------|------|----------|
> | 后端 | ? | ? |
> | 前端 | ? | ? |
> | 测试 | ? | ? |

用户回答后，生成 `resources.json` 并运行：
```bash
python3 <skill_dir>/scripts/plan_iteration.py <requirements.json> <resources.json>
```

将规划结果（各 Phase 的资源负载、溢出风险、建议调整）写入 PRD 第4章。

**如果用户说"不需要"或"跳过"**，则按默认规则自动推断：
- 按优先级分阶段：P0 → Phase 1，P1 → Phase 2，P2 → Phase 3
- 如果 Excel 有前端/后端/测试工时，自动计算各角色利用率
- 如果只有总工时，按需求类型自动估算拆分比例（纯后端 8:0:2，前后端 4:4:2，纯前端 0:8:2）
- 依赖关系从 Excel 依赖列提取

### Step 3：生成 PRD（后台子代理）

**生成大文档时不在主 session 中执行**，避免长时间阻塞对话。

流程：
1. 将解析结果（requirements.json + scan 结果 + plan 结果）写入临时文件
2. Spawn 子代理（推荐 `sessions_spawn`），把 PRD 模板 + 数据 + 项目文档路径一次性传入 task
3. 子代理后台独立生成，完成后自动通知
4. 主 session 立即回复用户："PRD 正在后台生成（预计 X 分钟），好了通知你"

子代理 task 模板：
```
你是专业产品经理。根据提供的需求数据和项目文档，按照 PRD 模板生成完整的需求文档。

输入：
- 需求数据：<requirements.json 的完整内容>
- 项目文档目录：<路径>（请自行读取）
- 迭代规划：<plan 结果>
- PRD 模板：<SKILL.md 中的文档结构和第6章模板>

输出文件：<输出路径>/PRD-{项目名}-v1.md

要求：
- 分段写入（每段 300-500 行），避免截断
- 第6章每条需求按模板完整展开
- 第10章从项目文档目录提取上下文
```

注意：
- 子代理模型选择：文档量大（>15条需求）用默认模型或高端模型，小文档可用轻量模型
- 不要用 `minimax-m2.5` 等轻量模型处理 >10 条需求的 PRD
- 子代理完成后，主 session 收到通知再执行 Step 4（联动询问）

输出文件：`<输出目录>/PRD-{项目名}-v1.md`

默认输出目录：Excel 同级目录下 `specs/`

### Step 4：跨技能联动（PRD 生成后）

PRD 生成完成后，**主动询问用户是否需要后续操作**：

> ✅ PRD 已生成！接下来要不要：
> 1. 📅 **拉个评审会？**（查空闲会议室 → 选参会人 → 订会议 → 建日程）
> 2. ✅ **按 Phase 1 拆任务分配？**（创建任务 → 分配负责人）
> 3. 📨 **发通知给相关人？**（IM / 邮件通知）

用户选择后，调用对应技能：
- **拉会议**：`oa-meeting`（rooms → book）+ `oa-schedule`（create + invite）+ `search-user`（选人）
- **拆任务**：`oa-task`（create，按 Phase 1 的需求逐条创建）
- **发通知**：IM 消息 / 邮件

**不强制**——用户说"不用"就结束。但必须**主动问一嘴**，不能默默完事。

## PRD 文档结构

```markdown
# PRD: {项目名} {版本} 迭代需求文档

> 版本 / 日期 / 作者 / 状态

## 修订记录
## 名词解释

## 第1章 产品概述
### 1.1 背景（需求产生的业务背景）
### 1.2 目标（本迭代要解决的核心问题）
### 1.3 成功指标（可量化）
### 1.4 涉及范围（做什么 + 不做什么）
### 1.5 阅读对象

## 第2章 产品结构图
### 2.1 功能结构图（模块→功能→子功能，标注 [NEW]/[增强]）
### 2.2 页面结构图（页面层级 + 新增页面标注）

## 第3章 功能需求清单
### 3.1 需求总览表（ID/模块/名称/优先级/后端工时/前端工时/测试工时/合计）
### 3.2 依赖关系

## 第4章 迭代计划
### 4.1 阶段计划（Phase 1/2/3 + 里程碑）
### 4.2 资源利用率（如有角色工时数据）
### 4.3 风险（概率/影响/应对）

## 第5章 业务流程
（核心功能的业务流程图 + 状态流转图，用文本流程图表示）

## 第6章 功能详情
（每条需求一个小节，按模板展开 — 见下方）

## 第7章 非功能需求（性能/安全/兼容性）
## 第8章 验收标准（功能验收 + 上线标准）
## 第9章 约束声明（DO NOT CHANGE 列表，告诉 AI 编码工具哪些不能改）

## 第10章 项目上下文
### 10.1 技术栈（语言/框架/构建工具/数据库/缓存等）
### 10.2 项目结构（核心模块列表 + 各模块职责）
### 10.3 构建与测试命令
### 10.4 现有数据库表（与本次需求相关的表名列表）
### 10.5 参考文档（项目文档路径列表，供 AI 编码工具按需读取）
```

## 第6章 每条需求模板

```markdown
### 6.N {ID}: {功能名称}

#### 用户故事
作为{角色}，我想要{功能}，以便{价值}。

#### 需求描述
（从 Excel 带入并补充完善）

#### 业务规则
（编号列表：核心逻辑 + 字段校验规则 + 权限规则 + 数量限制）

#### 数据模型
（新增/修改的表结构：表名、字段、类型、约束、默认值、校验、索引）

#### 交互说明
（ASCII 原型图 + 交互描述 + 页面状态：加载中/空状态/错误/权限不足）

#### 通知规则
（什么操作 → 通知谁 → 通知内容模板）

#### 异常/边界处理
（表格：网络异常/并发冲突/数据量大/权限不足/依赖不可用/边界值）

#### 验收标准
（表格：# / 场景 / 前置条件 / 操作 / 预期结果）
```

### 内容深度指南

| 需求类型 | 数据模型 | 交互说明 | 通知规则 | 验收场景数 |
|----------|----------|----------|----------|------------|
| 新功能（前后端） | 完整表结构+字段校验 | ASCII原型+全状态 | 完整 | 10-15 |
| 功能增强 | 新增/修改的字段 | 变更部分的交互 | 变更部分 | 8-12 |
| 纯后端优化 | 完整表结构 | 无或简略 | 无 | 5-10 |
| 纯前端优化 | 无 | 完整交互 | 无 | 5-8 |

## 脚本说明

| 脚本 | 用途 | 输入 | 输出 |
|------|------|------|------|
| `scripts/parse_excel.py` | 解析 Excel 需求表 | xlsx/xls/csv 路径 | JSON（需求列表+统计+依赖图） |
| `scripts/plan_iteration.py` | 迭代规划（可选） | JSON + 资源JSON | JSON（选中/溢出/风险/阶段） |
| `scripts/scan_project.py` | 扫描项目文档 | 目录路径 | JSON（技术栈/模块/构建命令） |

## 注意事项

1. **不写技术方案** — PRD 只描述"做什么"，不描述"怎么做"。技术方案由 AI 编码工具读完项目代码后自己设计。
2. **数据模型是需求的一部分** — 表结构、字段定义、校验规则属于需求描述，PRD 中应该包含。
3. **一个文件** — 所有需求在一份 PRD 中，不拆分。
4. **最少但必要的交互** — 共 3 个交互点：(1) 概览确认 (2) 团队配置 (3) PRD 后的联动询问。除此之外不要来回问。
5. **项目上下文写入第10章** — 用 `scan_project.py` 扫描结果填充。只列路径，不复制文档内容。
6. **大需求建议拆分** — 单条需求工时 > 8天时，在概览中提示用户考虑拆分。
7. **通用 skill** — 不预设任何项目或技术栈，Excel 列名自动检测。
8. **大文档后台生成** — 12条需求的 PRD 约 1500-2000 行，24条约 3000+ 行。**必须 spawn 子代理后台生成**，不在主 session 阻塞用户对话。子代理分段写入文件（每段 300-500 行），避免截断。
9. **背景信息来源** — 第1章产品概述的背景、目标等信息，Excel 里通常没有。agent 需要阅读项目文档目录中的文档（如需求说明、会议纪要等）来提取背景信息。如果项目文档也没有，则根据需求内容推断，并在概览中让用户确认。
10. **项目目录类型** — 用户给的可能是代码仓库（有 pom.xml/package.json 等），也可能是纯文档目录（Word/PDF/Markdown）。`scan_project.py` 两种都支持。代码仓库能提取技术栈和模块结构；纯文档目录主要提取文档列表供第10章引用。
11. **工时拆分** — 自动拆分比例仅作为初始估算。如果用户对拆分结果不满意，在概览确认时可以手动调整。实际项目中建议用户在 Excel 里直接填前端/后端/测试工时列。
12. **项目文档目录可选** — 没有也能生成 PRD，第10章项目上下文部分为空，第1章背景由 agent 根据需求内容推断。
